# 模块：聊天子视图与结果排序展示修复

## 已完成行为

- 数据问答页面的聊天工作区由内容驱动的最小高度改为视口受控高度。中间消息流是独立滚动区域，顶部会话标题和底部输入框固定在工作区内，长 SQL、图表和结果表不再拉长整页。
- 每条分析回复的 SQL、图表和结果表收敛为默认折叠的“查询详情”；展开后详情自身可滚动，结果表保留表内滚动，长会话仍由现有虚拟列表和向上加载机制处理。
- Presenter 不再反转 Executor 返回的行，严格保留 SQL `ORDER BY` 顺序；日期时间在 API 响应边界转为 ISO 字符串，在摘要中格式化为短日期。
- `month` 增加中文“月份”显示标签，月度查询摘要不再显示时区和零点时间。

## 根因与修复

截图 SQL 已按 `month ASC` 排序，数据库实际首行为 `2017-01-01 / 800`，末行为 `2017-12-01 / 5,673`。旧 Presenter 无条件执行 `reversed(execution.rows)`，导致摘要和图表/表格实际展示顺序与 SQL 相反。该反转已移除。

## API 与数据影响

- 未修改会话 API、SQL API、数据库结构或 SQL 安全链路。
- `AnalyzeResponse.rows` 的日期/时间值现在显式序列化为 ISO 字符串，满足既有 JSON 响应契约。
- 前端只消费既有 `summary`、`sql`、`rows`、`visualization` 字段，不引入 Mock 数据。

## 验证

- `python -m pytest backend/tests/test_analysis_presenter.py -q`：`5 passed`。
- `npm.cmd run frontend:build` 通过；仅有既有 bundle 大小提示。
- 真实 PostgreSQL 月度已支付订单 SQL 结果经 Presenter 回归：摘要为“首行月份为 2017-01-01，订单数为 800”，首行和末行分别为 2017-01 与 2017-12，范围为 2017-01-01 至 2017-12-01。
- 本机 Vite 服务已在 `http://127.0.0.1:5173` 启动。自动化浏览器环境无法访问宿主机本地端口，未能生成 UI 截图；需要在本机浏览器中确认独立滚动和详情展开交互。

## 剩余风险与后续

- 分析详情展开后由虚拟列表的动态测量更新高度；如果未来加入大规模富媒体结果，应为消息块增加更严格的高度预算和回归截图。
- 当前详情默认折叠，用户需要点击“查询详情”查看 SQL、图表和完整结果；这是为保证长期会话可扫描性做出的取舍。
