# 聊天子视图与结果排序展示修复计划

## Goal

将问答页面的会话流改造成独立、可滚动的聊天子视图，避免 SQL、图表和结果表把整个聊天页面无限拉长；同时修复按月升序 SQL 结果被摘要错误展示为末行的问题。

## Scope

- 保持会话列表、新建会话、分页加载、流式状态和输入框功能，重构中间聊天区域为受控高度的独立滚动视图。
- 将分析结果以消息内的可折叠详情或独立结果区域呈现，默认不撑开整个会话流；聊天可向下滚动查看历史。
- 统一结果行顺序、首行摘要和日期显示，保证后端 `ORDER BY` 顺序不被前端或 Presenter 改写。
- 为结果摘要/日期格式补充后端测试，并做前端生产构建与真实查询验证。

## Out of scope

- 不修改 SQL 生成、Query Plan、Guard、Executor 或数据库业务数据。
- 不更改会话 API 分页契约，不增加 Mock 数据。
- 不重新设计全站导航或引入新的前端框架。

## Implementation steps

- [x] 定位聊天页的滚动容器、结果卡片与摘要生成的排序/日期处理问题。
- [x] 将聊天消息区重构为独立受控高度子视图，保留自动滚动、向上加载和底部输入。
- [x] 收敛分析结果展示高度，支持在会话中展开查看 SQL、图表和表格。
- [x] 修复首行摘要与日期格式，补充对应测试。
- [x] 执行前端构建、focused 后端测试和真实已支付订单月度查询回归。

## Validation plan

- 后端 Presenter/Result Contract focused pytest，断言按月升序的首行摘要为 2017-01 且日期不携带时区 ISO 串。
- `npm.cmd run frontend:build`。
- 使用浏览器检查桌面与窄屏下聊天区独立滚动、长 SQL 不挤占输入区、结果可展开。
- 真实认证查询对照 SQL 返回的首行与 UI 摘要一致。

## Risks

- 虚拟列表、分页加载和独立滚动容器的滚动锚点可能互相影响，必须覆盖加载旧消息和流式消息两类状态。
- 图表、SQL 和结果表的高度收敛不能隐藏执行状态、错误或关键结果。
